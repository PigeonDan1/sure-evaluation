# Pipeline Catalog

This directory contains a machine-readable catalog of every metric pipeline that `sure-eval metric describe` can produce.

## File: `pipeline_catalog.jsonl`

Each line is one JSON object describing one metric pipeline. The file is
generated by running `sure-eval metric describe` for every supported task,
language, metric, exact pipeline-ID variant, and method-specific selector
combination.

### Schema

| Field | Type | Description |
|:------|:-----|:------------|
| `task` | string | Canonical task name (e.g. `ASR`, `TTS`) |
| `task_alias` | string | CLI task argument (e.g. `asr`, `tts`) |
| `language` | string | Language profile, or `n/a` for language-independent tasks |
| `metric` | string | Canonical reported metric, such as `cer`, `wer`, `spk_sim`, `si_sdr`, `wv_mos`, `macro_recall`, or `multi` |
| `pipeline_id` | string | Atomic or bundle pipeline identity |
| `pipeline_kind` | string | `atomic` for one metric, `bundle` for multi-metric selections |
| `member_pipeline_ids` | list | Atomic member pipeline IDs for bundle rows |
| `execution_metrics` | list | Executor-facing selectors used to run the pipeline, such as `tts_cer`, `sim/wavlm-large`, `wv-mos`, or `macro-recall` |
| `nodes` | list | Deduplicated selected runtime node IDs |
| `computation_node_ids` | list | Ordered score-affecting computation nodes; conversions are included |
| `task_config_path` | string | Relative path to the task manifest |
| `route_config_path` | string | Relative path to the task route declaration |
| `describe_entrypoint` | string | Dotted Python entrypoint used by `metric describe` |
| `script_entrypoint` | string | Dotted Python entrypoint used by `metric run` |
| `executor` | string | Dotted task executor called by the script route |
| `required_roles` | list | Input roles you must provide to `sure-eval metric run` |
| `optional_roles` | list | Input roles you may optionally provide |

### Example row

```json
{
  "task": "ASR",
  "task_alias": "asr",
  "language": "zh",
  "metric": "cer",
  "pipeline_id": "asr.zh.cer.wetext_norm_zh_itn_v1.wenet_cer_v1",
  "pipeline_kind": "atomic",
  "member_pipeline_ids": [],
  "execution_metrics": ["cer"],
  "nodes": ["normalization/wetext_norm", "scoring/wenet_cer"],
  "computation_node_ids": ["normalization/wetext_norm", "scoring/wenet_cer"],
  "task_config_path": "src/sure_eval/evaluation/tasks/asr/manifest.yaml",
  "route_config_path": "src/sure_eval/evaluation/tasks/asr/routes.yaml",
  "describe_entrypoint": "sure_eval.evaluation.scripts.asr.describe_pipeline",
  "script_entrypoint": "sure_eval.evaluation.scripts.asr.run",
  "executor": "sure_eval.evaluation.tasks.asr.pipeline.evaluate_asr_files",
  "required_roles": ["hyp", "ref"],
  "optional_roles": []
}
```

## How to read the catalog

Find the metric you want to run, then look at `nodes` and `required_roles`:

- `nodes` tells you which evaluation nodes are executed.
- `required_roles` tells you which CLI arguments or Python API fields are mandatory.

For example, `s2tt.zh.xcomet_xl.xcomet_xl_v1` requires `src`, `hyp`, and `ref`:

```json
{
  "task": "S2TT",
  "task_alias": "s2tt",
  "language": "zh",
  "metric": "xcomet_xl",
  "pipeline_id": "s2tt.zh.xcomet_xl.xcomet_xl_v1",
  "pipeline_kind": "atomic",
  "member_pipeline_ids": [],
  "execution_metrics": ["xcomet_xl"],
  "nodes": ["scoring/xcomet_xl"],
  "computation_node_ids": ["scoring/xcomet_xl"],
  "task_config_path": "src/sure_eval/evaluation/tasks/s2tt/manifest.yaml",
  "route_config_path": "src/sure_eval/evaluation/tasks/s2tt/routes.yaml",
  "describe_entrypoint": "sure_eval.evaluation.scripts.s2tt.describe_pipeline",
  "script_entrypoint": "sure_eval.evaluation.scripts.s2tt.run",
  "executor": "sure_eval.evaluation.tasks.s2tt.pipeline.evaluate_s2tt_files",
  "required_roles": ["src", "hyp", "ref"],
  "optional_roles": []
}
```

## How to regenerate

```bash
python scripts/generate_pipeline_catalog.py
```

The script runs `describe_pipeline` for every supported combination and writes `docs/pipeline_catalog.jsonl`.

## Notes

- `pipeline_id` uses `task.language.metric.node_version...`. Language-independent tasks use `any`.
- Multi-metric audio selections are bundle rows, such as `tts.zh.multi.cer.funasr_loader_16k_mono_v1.paraformer_zh_v1.punctuation_strip_norm_v1.wenet_cer_v1__spk_sim.wavlm_large_sim_v1__dnsmos.dnsmos_v1`, with atomic member IDs in `member_pipeline_ids`.
- `execution_metrics` records selector compatibility and method selection. For example, a TTS CER pipeline reports `metric=cer` and `execution_metrics=["tts_cer"]`; a WavLM speaker-similarity pipeline reports `metric=spk_sim` and `execution_metrics=["sim/wavlm-large"]`. ASR route variants with the same metric are selected by exact `pipeline_id`.
- KWS file pipelines include conversion nodes in `pipeline_id`, such as `kws.any.macro_recall.conversion_kws_sure_json_to_samples_v1.wekws_det_v1`.
- Paths in the catalog are repository-relative, never absolute host paths.
- Node-local environments are required for heavy nodes such as `scoring/xcomet_xl`, `scoring/dnsmos`, and `transcription/whisper_large_v3`. Optional pip packages are required for nodes such as `scoring/stoi` and `scoring/pesq`. Use `sure-eval env setup --node <node>` to prepare them.
